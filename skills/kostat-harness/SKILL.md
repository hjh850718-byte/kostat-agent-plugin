# kostat-harness — 에이전트팀 하네스 메타 스킬

## 정체성

이 스킬은 **하네스 빌더(Harness Builder)** 다.
사용자의 도메인 설명 한 마디("딥 리서치 하네스 구성해줘")를 입력받아:

1. 도메인에 맞는 전문 에이전트팀 아키텍처를 설계하고
2. 에이전트 정의 파일 (`.claude/agents/*.md`)과 스킬 파일 (`skills/*/SKILL.md`)을 자동 생성하며
3. 팀 간 오케스트레이션과 에러 핸들링 프로토콜을 완성한다

**핵심 원칙**: 생성된 파일은 즉시 Claude Code에서 사용 가능한 프로덕션 레디 상태여야 한다.
에이전트 팀 기능(`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`) 환경에서 실행된다.

---

## 트리거 조건

다음 패턴이 감지되면 즉시 이 스킬을 실행한다:

| 트리거 패턴 | 예시 |
|-------------|------|
| `하네스 구성해줘` + 도메인 설명 | "딥 리서치 하네스 구성해줘" |
| `harness` + 도메인 설명 | "코드 리뷰 harness 만들어줘" |
| `에이전트팀 만들어줘` + 목적 | "마케팅 에이전트팀 만들어줘" |
| `팀 구성` + 도메인 설명 | "웹사이트 제작 팀 구성해줘" |
| `하네스를 구성` + 역할 나열 | "리서치, 분석, 보고서 작성 에이전트 하네스를 구성해줘" |

**항상 6단계 워크플로우 전체를 완주한다.** 중간에 멈추지 않는다.

## 아키텍처 패턴 6가지

### 패턴 1 — 파이프라인 (Pipeline)

```
[입력]
  │
  ▼
Agent-A (1단계)
  │ 출력 → 다음 입력
  ▼
Agent-B (2단계)
  │ 출력 → 다음 입력
  ▼
Agent-C (3단계)
  │
  ▼
[최종 출력]
```

**선택 기준**: 각 단계가 이전 단계 결과에 의존하는 순차 작업
**예시 도메인**: API 문서 생성 (분석→작성→검토→배포), 영상 편집 (촬영분석→편집→자막→업로드)
**에이전트 수**: 3~5개
**제약**: 한 에이전트가 실패하면 파이프라인 전체 중단 → 에러 핸들링 필수

---

### 패턴 2 — 팬아웃/팬인 (Fan-out / Fan-in)

```
[공통 입력]
      │
  ┌───┼───┐
  ▼   ▼   ▼
A-1  A-2  A-3   ← 독립 병렬 실행 (컨텍스트 공유 금지)
  │   │   │
  └───┼───┘
      ▼
  Aggregator
  (결과 취합)
      │
      ▼
  [통합 출력]
```

**선택 기준**: 동일 입력을 다른 관점에서 병렬 처리, 독립 실행 가능
**예시 도메인**: 코드 리뷰(보안/성능/스타일 동시), 딥 리서치(웹/학술/커뮤니티 동시), 번역(다국어 동시)
**에이전트 수**: 2~5개 (병렬) + 1개 (Aggregator)
**제약**: 병렬 에이전트들은 서로 컨텍스트를 공유하면 안 됨 (독립 실행 강제)

---

### 패턴 3 — 전문가 풀 (Expert Pool)

```
[요청 입력]
      │
      ▼
  Router
  (분류/판단)
  │   │   │
  ▼   ▼   ▼
Expert Expert Expert
  -법률  -기술  -재무
    │
    ▼ (해당 전문가만 호출)
  [전문 응답]
```

**선택 기준**: 입력 유형에 따라 다른 전문가가 필요한 경우, 전문가 중 1~2명만 실제 호출
**예시 도메인**: 고객 문의 분류(기술/환불/계정), 법무 검토(계약/특허/규정), 의료 상담(내과/외과/정신건강)
**에이전트 수**: 1개 (Router) + 3~6개 (전문가 풀)
**제약**: Router가 잘못 분류하면 전체 품질 저하 → Router 프롬프트 정교화 필수

---

### 패턴 4 — 생성-검증 (Gen-Eval Loop)

```
[입력]
  │
  ▼
Generator  ← 초안 생성
  │
  ▼
Evaluator  ← 독립 검증 (Generator와 컨텍스트 분리)
  │
  ├─ Pass → [최종 출력]
  │
  └─ Fail → 피드백 → Generator (재생성, 최대 3회)
               ↑__________________________|
```

**선택 기준**: 출력 품질이 중요하여 독립 검증이 필요한 경우
**예시 도메인**: 인보이스/재무 계산(금액 오류 불허), 계약서 초안(법적 검토 필수), 코드 생성(테스트 통과 필요)
**에이전트 수**: 1개 (Generator) + 1개 (Evaluator)
**제약**: Generator ≠ Evaluator (동일 컨텍스트 금지), Loop 최대 3회 후 에스컬레이션

---

### 패턴 5 — 감독자 (Supervisor)

```
[목표 입력]
      │
      ▼
  Supervisor  ← 동적 계획 수립 + 우선순위 결정
  │   │   │
  ▼   ▼   ▼
Agent Agent Agent  ← 감독자가 동적으로 배정
  │
  ▼
  Supervisor  ← 진행 상태 모니터링 + 재배정
  │
  ▼
[완성 결과]
```

**선택 기준**: 목표가 복잡하고 작업 흐름이 실행 중에 변하는 경우, 중앙 조율이 필요
**예시 도메인**: 소프트웨어 프로젝트 관리, 마케팅 캠페인 전체 조율, 연구 프로젝트 운영
**에이전트 수**: 1개 (Supervisor) + 3~6개 (실행 에이전트)
**제약**: Supervisor가 과부하되면 전체 속도 저하 → Supervisor는 계획/위임만, 직접 실행 금지

---

### 패턴 6 — 계층적 위임 (Hierarchical Delegation)

```
[최상위 목표]
      │
      ▼
  L1-Orchestrator
  │           │
  ▼           ▼
L2-Manager  L2-Manager
  │   │       │   │
  ▼   ▼       ▼   ▼
L3  L3       L3  L3  ← 말단 실행 에이전트
```

**선택 기준**: 매우 큰 목표를 계층별로 분해, 각 레이어가 독립 책임
**예시 도메인**: 대규모 콘텐츠 제작(플랫폼별 팀), 엔터프라이즈 자동화, 멀티 제품 출시 조율
**에이전트 수**: 7~12개 (계층 구조)
**제약**: 계층 깊이는 최대 3레벨로 제한 (깊을수록 복잡도·지연 증가)

## 실행 모드 판단

하네스를 생성할 때 두 가지 실행 모드 중 하나를 선택한다.

### 판단 기준

```
에이전트 수가 2개 이상인가?
    AND
에이전트 간 결과 공유/협업이 필요한가?
    AND
작업이 단발성이 아닌가?
        │
    YES → 에이전트 팀 모드
    NO  → 서브 에이전트 모드
```

### 에이전트 팀 모드 (기본 — 협업 필요 시)

**사용 도구**: `TeamCreate` + `SendMessage` + `TaskCreate`

**생성 파일**:
- `.claude/agents/{agent-name}.md` — 각 에이전트 정의
- `skills/{harness-name}-orchestrator/SKILL.md` — 팀 오케스트레이터

**적합한 패턴**: 팬아웃/팬인, 전문가 풀, 감독자, 계층적 위임

**오케스트레이터 스킬에서 팀 실행 방법**:

```markdown
## 팀 실행 시퀀스

1. TaskCreate로 각 에이전트 작업 등록
2. 병렬 에이전트는 동시에 TaskCreate 호출
3. 각 Task 완료 시 SendMessage로 Aggregator에 결과 전달
4. Aggregator가 최종 결과 취합
```

### 서브 에이전트 모드 (단발성 · 통신 불필요 시)

**사용 도구**: `Agent()` 직접 호출

**생성 파일**:
- `skills/{harness-name}/SKILL.md` — Agent() 호출 로직 포함

**적합한 패턴**: 파이프라인, 생성-검증 (단순 케이스)

## 6단계 워크플로우

### STEP 1 — 도메인 분석

도메인 설명에서 다음 항목을 추출한다:

```
분석 항목:
- 핵심 목표: [무엇을 달성하는가]
- 주요 작업 유형: [어떤 종류의 작업들인가]
- 작업 간 의존성: [순차/병렬/조건부]
- 전문 지식 분야: [어떤 도메인 지식이 필요한가]
- 품질 요구사항: [검증/검토/승인 필요 여부]
- 출력 형식: [파일/텍스트/API 응답/알림]
- 반복성: [일회성/반복/자동화]
```

**출력**: 도메인 분석 요약 + 권장 패턴 후보 (2~3개)

---

### STEP 2 — 팀 아키텍처 설계

STEP 1 분석 결과를 바탕으로:

1. **패턴 확정**: 6가지 패턴 중 가장 적합한 1개 선택 + 선택 이유 1줄 설명
2. **실행 모드 확정**: 에이전트 팀 vs 서브 에이전트 + 판단 근거
3. **에이전트 목록 정의**:

```markdown
| 에이전트 이름 | 역할 | 입력 | 출력 | 의존 에이전트 |
|--------------|------|------|------|--------------|
| {name}       | [역할] | [입력] | [출력] | [없음/에이전트명] |
```

4. **아키텍처 다이어그램** (ASCII): 실제 흐름 시각화

**이 단계 완료 전까지 파일 생성 금지.**

---

### STEP 3 — 에이전트 정의 생성

에이전트 팀 모드: `.claude/agents/` 디렉토리에 에이전트 정의 파일 생성.
서브 에이전트 모드: 에이전트 정의는 스킬 파일 내부에 포함.

#### 에이전트 정의 파일 템플릿 (`.claude/agents/{agent-name}.md`)

```markdown
---
name: {agent-name}
description: "{역할 한 줄 — 언제 호출되는지 구체적으로 명시}"
---

# {Agent Display Name}

## 역할
{이 에이전트가 담당하는 책임 범위 — 구체적으로}

## 전문 분야
{처리할 수 있는 도메인 지식 또는 기술 스택}

## 입력 스펙
- 필수: {필수 입력 항목}
- 선택: {선택 입력 항목}
- 형식: {JSON / 파일 경로 / 텍스트}

## 작업 순서
1. {1단계: 입력 검증}
2. {2단계: 핵심 처리}
3. {3단계: 출력 형식화}

## 출력 스펙
{출력 형식 정의 — JSON 스키마 또는 마크다운 템플릿}

## 제약 사항
- {금지 행동 1}
- {금지 행동 2}
- 실패 시: {에러 처리 방법}

## 다음 에이전트 핸드오프
{결과를 전달할 다음 에이전트 이름 또는 "없음 (최종 출력)"}
```

**작성 규칙**:
- `description`은 Router/Supervisor가 에이전트 선택 시 읽는 핵심 정보 — 명확하게
- 입력/출력 스펙은 JSON 형식으로 구체화
- 제약 사항에 "하지 말아야 할 것"을 반드시 포함
- 에이전트 이름: `{harness-prefix}-{role}` 형식 (예: `research-web-searcher`)

---

### STEP 4 — 스킬 생성

각 에이전트가 사용할 스킬 파일을 `skills/` 디렉토리에 생성한다.

#### 스킬 파일 템플릿 (`skills/{harness-name}-{role}/SKILL.md`)

```markdown
---
name: {harness-name}-{role}
description: "{역할 설명}. 트리거: {키워드 목록}"
---

# {Skill Display Name}

## 실행 모드

| 모드 | 동작 | 사용 시나리오 |
|------|------|-------------|
| **{pattern-mode}** (기본) | {패턴 방식 설명} | {Orchestrator/팀에서 호출 시} |
| **standalone** | {단독 실행 방법} | {사용자 직접 호출 시} |

## {패턴 모드} 워크플로우

{패턴에 맞는 실행 흐름 다이어그램}

## 세부 작업 순서

### {주요 단계 1}
{구체적인 처리 지침}

### {주요 단계 2}
{구체적인 처리 지침}

## 출력 형식

```json
{
  "status": "ok | error | partial",
  "result": {},
  "next_agent": "{다음 에이전트 이름 또는 null}",
  "error": null
}
```

## 검증 체크리스트
- [ ] {필수 검증 항목 1}
- [ ] {필수 검증 항목 2}
- [ ] 출력 JSON 스키마 일치 확인
- [ ] 다음 에이전트 핸드오프 데이터 준비 완료

## 에러 처리
| 에러 유형 | 처리 방법 |
|----------|----------|
| 입력 누락 | {처리 방법} |
| 처리 실패 | 재시도 1회 → 실패 시 에스컬레이션 |
| 타임아웃 | 10분 초과 시 부분 결과로 처리 |
```

---

### STEP 5 — 통합 및 오케스트레이션

하네스 전체를 조율하는 오케스트레이터 스킬을 생성한다.

#### 오케스트레이터 스킬 템플릿 (`skills/{harness-name}-orchestrator/SKILL.md`)

```markdown
---
name: {harness-name}-orchestrator
description: "{하네스 이름} 오케스트레이터 — 트리거 감지 → 에이전트팀 소집 → 결과 취합"
---

# {Harness Name} Orchestrator

## 정체성
{도메인}의 {목표}를 달성하기 위한 에이전트팀 조율자.
선택된 아키텍처 패턴: {패턴 이름}
실행 모드: {에이전트 팀 | 서브 에이전트}

## 트리거 테이블

| 트리거 | 우선순위 | 소집 에이전트 | 패턴 |
|--------|---------|-------------|------|
| {트리거 1} | 1 | {에이전트 목록} | {패턴} |
| {트리거 2} | 2 | {에이전트 목록} | {패턴} |

## 실행 흐름

{선택된 패턴의 실행 다이어그램}

## 에이전트 팀 구성

| 에이전트 | 스킬 | 투입 조건 | 실행 순서 |
|----------|------|----------|----------|
| {에이전트 이름} | {스킬 이름} | {조건} | {순서/병렬} |

## 실행 모드별 호출 방법

### 에이전트 팀 모드 (TeamCreate 방식)
```
각 에이전트 Task를 동시에 TaskCreate로 등록
→ 병렬 에이전트 완료 대기
→ SendMessage로 Aggregator에 결과 전달
→ Aggregator 최종 결과 취합 후 출력
```

### 서브 에이전트 모드 (Agent() 방식)
```
Agent(analyst) → 결과1
Agent(builder) → 결과2
결과1 + 결과2 → Agent(reviewer) → 최종 결과
```

## 에러 핸들링 프로토콜

| 에러 유형 | 처리 방법 |
|----------|----------|
| 단일 에이전트 실패 | 재시도 1회 → 실패 시 해당 에이전트 스킵 + 알림 |
| 2개 이상 에이전트 실패 | 전체 중단 → 긴급 알림 |
| Validator 에러 발견 | 사용자 확인 요청 (자동 진행 금지) |
| 타임아웃 (10분) | 강제 종료 → 부분 결과로 처리 |
| 의존 에이전트 실패 | 후속 에이전트 스킵 + 부분 결과 |

## 모니터링

```
pending → running → completed
                 → failed → retry → failed_final
                 → timeout
```

- Task 타임아웃: 10분
- 재시도: 1회
- 동시 실행 최대: 4개 Task
- 전체 팀 타임아웃: 15분

## 출력 형식

```json
{
  "harness": "{harness-name}",
  "pattern": "{아키텍처 패턴}",
  "trigger": "{감지된 트리거}",
  "agents_executed": [
    {"name": "{에이전트}", "status": "completed", "duration_sec": 0}
  ],
  "result": {},
  "errors": [],
  "escalation_needed": false
}
```
```

### STEP 6 — 검증 및 테스트

모든 파일 생성 완료 후 아래 체크리스트를 실행한다.

#### 구조 검증

- [ ] `.claude/agents/` 에 에이전트 정의 파일이 모두 존재하는가
- [ ] 각 에이전트 파일에 `name`, `description` frontmatter가 있는가
- [ ] `skills/{harness-name}-orchestrator/SKILL.md`가 존재하는가
- [ ] 각 에이전트 스킬 파일이 `skills/` 에 존재하는가

#### 내용 검증

- [ ] 오케스트레이터의 에이전트 목록과 실제 생성된 에이전트 수가 일치하는가
- [ ] 입력/출력 스펙이 에이전트 간 핸드오프 체인에서 호환되는가
- [ ] 에러 핸들링 프로토콜이 오케스트레이터와 각 스킬에 정의되어 있는가
- [ ] 선택된 패턴의 다이어그램이 실제 에이전트 흐름과 일치하는가
- [ ] 생성-검증 패턴: Generator/Evaluator가 컨텍스트 분리 지침을 포함하는가
- [ ] 팬아웃 패턴: 병렬 에이전트들이 "컨텍스트 공유 금지" 제약을 포함하는가

#### 호환성 검증

- [ ] 에이전트 팀 모드: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 환경 변수 요구 명시
- [ ] 서브 에이전트 모드: `Agent()` 호출 형식이 올바른가
- [ ] Plugin과의 충돌 없음 (기존 kostat 스킬 이름과 중복 없음)

검증 실패 항목이 있으면 즉시 수정 후 재검증한다 (최대 2회 반복).

## 에러 핸들링 공통 프로토콜

모든 하네스에 동일하게 적용되는 표준 에러 처리 테이블:

| 에러 유형 | 심각도 | 처리 방법 | 사용자 알림 |
|----------|--------|----------|------------|
| 에이전트 입력 누락 | 중간 | 기본값 적용 또는 스킵 | 경고 |
| 단일 에이전트 실패 | 중간 | 재시도 1회 → 스킵 | 알림 |
| Evaluator 검증 실패 | 높음 | Generator 재실행 (최대 3회) | 진행 상태 |
| 2개 이상 에이전트 실패 | 높음 | 전체 중단 | 긴급 알림 |
| 타임아웃 (10분) | 높음 | 강제 종료 → 부분 결과 | 경고 |
| API 키 없음 | 중간 | 해당 에이전트 스킵 | 설정 안내 |
| 파일 접근 불가 | 높음 | 전체 중단 | 에러 보고 |
| 루프 최대 초과 (3회) | 높음 | 사용자 에스컬레이션 | 수동 처리 요청 |

**에스컬레이션 규칙**:
- 자동 재시도로 해결 불가 → 즉시 사용자에게 보고
- 사용자 판단이 필요한 결정은 자동 진행 금지
- 에스컬레이션 메시지: 에러 유형 + 현재 상태 + 권장 조치 포함

## 제약 사항

- 하네스 생성 시 기존 KOSTAT 스킬 (`kostat-*`) 이름과 중복되는 스킬 생성 금지
- 에이전트 파일은 반드시 `name`과 `description` frontmatter를 포함해야 함
- 오케스트레이터 스킬은 반드시 에러 핸들링 프로토콜 섹션을 포함해야 함
- 생성-검증 패턴에서 Generator와 Evaluator는 동일 컨텍스트 실행 금지 지침 포함 필수
- 팬아웃 병렬 에이전트들은 "컨텍스트 공유 금지" 명시 필수
- 6단계 워크플로우를 모두 완주해야 하며, 각 단계 결과를 사용자에게 보고 후 다음 단계로 진행
- STEP 2 아키텍처 설계 확인 전 STEP 3 파일 생성 시작 금지
