## 2. 발견된 리스크 & 개선점

### 2.1 🔴 Critical: Excel 파일 경로 의존성

**문제:** Agent Container 내부에서 `kostat-excel-mcp`가 Excel 파일에 접근해야 함.
현재 `kostat_excel_mcp.py`는 로컬 Windows 경로 하드코딩.

**해결 방안:**
```
옵션 A — File Upload (권장):
  1. Python으로 Excel을 미리 업로드: client.beta.files.upload()
  2. Session 생성 시 resources에 mount
  3. MCP 서버가 mount_path(/workspace/data.xlsx)를 읽도록 env 전달

옵션 B — MCP 서버가 직접 파일 서빙:
  1. MCP 서버가 실행되는 호스트에 Excel 파일 저장
  2. KOSTAT_EXCEL_PATH 환경변수로 경로 주입
  3. Agent Container는 MCP를 통해 간접 접근 (직접 파일 읽기 불가)
```

**권장:** 옵션 A. Agent가 bash로 직접 Excel을 읽고 쓸 수 있어 유연성 높음.
단, 파일 업로드는 Session 생성 시점에 필요하므로 `run_kostat_agent.py`에서 처리.

### 2.2 🟠 Medium: MCP 서버 호스팅

**문제:** `kostat-excel-mcp`와 `kostat-email-mcp`를 SSE 엔드포인트로
노출해야 Agent가 접근 가능. 현재 로컬 stdio 모드만 구현됨.

**해결 방안:**
```
1. ngrok/Cloudflare Tunnel로 로컬 MCP 서버 임시 노출 (개발/데모)
2. VPS/클라우드에 MCP 서버 배포 + HTTPS + 인증
3. 또는 odysseus가 이미 MCP 관리 기능 보유 (app.py line 664-671)
   → odysseus 내장 MCP Manager 활용 가능
```

**odysseus MCP Manager 참고** (`app.py` 664-671):
```python
from src.mcp_manager import McpManager
mcp_manager = McpManager()
# ... mcp_routes 등록
# 이미 MCP 서버 연결 관리 기능 내장
```

**권장:** 단기 = odysseus의 MCP Manager 활용. 장기 = 전용 MCP 서버 호스팅.

### 2.3 🟠 Medium: Custom Tool 처리의 동기화 문제

**문제:** `update_po()` custom tool이 Agent → App → Excel 기록 순서로 동작.
사용자가 "확인" 메시지를 보내면 Agent가 tool 호출, App이 Excel에 기록.
그런데 사용자 확인과 tool 호출 사이에 **Agent의 추가 추론**이 발생할 수 있음
(예: "이 내용이 맞는지 다시 검증" → 사용자 혼란).

**해결 방안:**
```
system prompt에 명시:
  "사용자 확인을 받은 후 custom tool을 호출할 때는 추가 질문 없이
   즉시 tool_use 이벤트를 발생시키세요. 사용자가 이미 확인했습니다."
```

### 2.4 🟡 Low: Stateful 세션 vs Stateless 요청

**문제:** Managed Agent Session은 상태를 유지하지만(대화 기록, 파일 상태),
`run_kostat_agent.py`는 매 실행 새 Session 생성. 이전 Session의 맥락이 유실됨.

**해결 방안:**
```
옵션 A — Memory Store 사용:
  Agent 생성 시 Memory Store 연결 → cross-session memory 유지
  예: "저번에 봤던 Skyworks PO 현황 기억나?" 가능

옵션 B — Session 재사용:
  idle 상태의 Session에 새 message 전송 (연속 작업에 적합)
  단, Session은 영구적이지 않음 (archive/delete 가능)

옵션 C — 매번 신규 Session (현재 방식):
  간단하고 예측 가능. 이전 맥락이 필요하면 사용자가 직접 설명.
```

### 2.5 🟡 Low: Excel 동시 쓰기 충돌

**문제:** Agent가 `update_po_field()` MCP tool로 Excel을 직접 수정 가능.
2개 Session이 동시에 같은 Excel을 수정하면 데이터 충돌.

**해결 방안:**
```
1. DRY_RUN=true 유지 (MCP 직접 쓰기 차단)
2. 모든 쓰기는 Custom tool(update_po)로만 → App이 단일 파일 Lock 관리
3. Python의 threading.Lock 또는 file lock(LockFile) 사용
```

### 2.6 🟡 Low: 대용량 Excel 처리

**문제:** OOR/커미션 데이터가 1000행 이상일 경우
`_sheet_to_text()`가 50행 제한으로 전체 데이터를 못 보여줌.

**해결 방안:**
```
1. MCP tool에 max_rows 파라미터 추가 (완료)
2. Agent가 "전체 데이터 보여줘" 요청 시 max_rows=200으로 재조회
3. 필요시 CSV 변환 후 bash로 분석 (agent_toolset 활용)
```

## 4. 개선 권장사항 (우선순위 순)

### P0 — 해야 함 (안 하면 동작 안 함)

1. **Excel 파일을 Session Resource로 마운트**
   - `run_kostat_agent.py`에서 `client.beta.files.upload()` → `sessions.create(resources=[...])`
   - MCP 서버가 mount_path를 읽도록 env 전달

2. **MCP 서버 SSE 호스팅 결정**
   - ngrok으로 빠르게 테스트할지, odysseus MCP Manager로 할지, VPS에 할지
   - 결정 후 `kostat.agent.yaml`의 `mcp_servers.url` 업데이트

### P1 — 권장 (사용성/안정성)

3. **Multi-agent roster 추가** — kostat-po-agent(parsing) + kostat-report-agent(report) 위임
4. **Vault credential 구체화** — MCP OAuth 토큰 발급 절차 문서화
5. **Stream 재접속 로직** — SSE 유실 시 events.list()로 복구

### P2 — 고려 (확장성)

6. **Memory Store 연결** — 고객사별 선호도/이력 유지
7. **Notion MCP 연동** — 고객 커뮤니케이션 로그 자동 저장
8. **Session output 수집** — Agent가 생성한 파일 자동 다운로드

## 5. 다이어그램: 현재 vs 권장 아키텍처

### 현재 설계:
```
사용자 → Python SDK → CMA Session (1회성)
                         ├─ agent_toolset (8종)
                         ├─ MCP: kostat-excel (읽기+DRY_RUN 쓰기)
                         ├─ MCP: kostat-email (Outlook)
                         └─ Custom: update_po, send_email
```

### 권장 (P0 반영):
```
사용자 → Python SDK → CMA Session (1회성)
                         ├─ agent_toolset (8종)
                         ├─ MCP: kostat-excel (읽기 전용)
                         │   └─ Excel File mounted at /workspace/
                         ├─ MCP: kostat-email (Outlook)
                         ├─ MCP: kostat-notion (고객 로그)
                         ├─ Multi-agent:
                         │   ├─ po-agent (PDF 파싱 전담)
                         │   └─ report-agent (리포트 작성 전담)
                         ├─ Memory Store: cross-session context
                         └─ Custom: update_po, send_email
```

## 6. 결론

| 영역 | 평가 |
|------|------|
| **Console 설계** | ✅ 적합. Agent YAML로 버전 관리 가능 |
| **System Prompt** | ✅ KOSTAT 업무 4대 영역 + 행동 원칙 명확 |
| **Tools/MCP 구성** | ⚠️ MCP 호스팅 방식 결정 필요, 파일 마운트 필요 |
| **SDK 호출 코드** | ✅ 기본 동작. 파일 업로드 + Stream 복구 보강 필요 |
| **Security** | ⚠️ DRY_RUN 모드로 안전하나, 실제 배포 시 인증/암호화 필요 |
| **확장성** | ✅ Multi-agent/Memory/Notion 연동 가능 구조 |

### 핵심 발견 요약

1. **Excel 파일 접근 방식**이 가장 큰 설계 결정 — File Upload mount vs MCP 간접 접근
2. **MCP 서버 호스팅**이 실제 운영의 병목 — 로컬 개발과 배포 환경 분리 필요
3. **Custom tool의 역할 명확** — 데이터 변경은 Custom tool로만, MCP는 읽기 전용
